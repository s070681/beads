# Expand /beads:onboard v3

## Overview

Enhance the `/beads:onboard` command (385 lines → ~500-550 lines) to teach agents the
*practice* of using beads well, not just structural conventions. Adds three-way project
mode detection, adaptive re-onboard via persisted state, and five new quality teaching
sections in the generated reference.md. All changes are to one file:
`claude-plugin/commands/onboard.md` and its embedded templates.

No Go code changes. No new files beyond what onboard generates.

## Problem Statement / Motivation

Empirical analysis of 354 beads from the portfolio project (see brainstorm:
`docs/brainstorms/2026-03-04-onboard-v3-brainstorm.md`) revealed that:

1. Agents don't structure bead descriptions unless told which sections to use (14% → 36%
   with guidance, still inconsistent)
2. Agents dump everything into `bd comments` instead of using structured fields
   (`--design`, `--notes`, `--acceptance`)
3. Priority assignment is inconsistent without project-specific calibration
4. Onboard can't handle new/empty projects (nothing to detect from)
5. Re-running onboard after the command improves overwrites customizations

SkillsBench research (arxiv.org/abs/2602.12670) confirms: agents can't reliably author
structured content from abstract rules. They need concrete triggers, filled-in examples,
and decision tests. "Use when relevant" guidance gets skipped.

## Proposed Solution

Modify `claude-plugin/commands/onboard.md` in 7 implementation steps. Each step modifies
a specific phase of the command. The interview stays at 6 questions (Q1 gets richer). The
expansion goes primarily into the reference.md template.

## Orientation

**Primary file:** `claude-plugin/commands/onboard.md` (the only file being modified)

**Reference files to read before starting:**
- `docs/brainstorms/2026-03-04-onboard-v3-brainstorm.md` — research data
- `docs/brainstorms/2026-03-04-onboard-v3-design-decisions.md` — all design decisions
- `docs/brainstorms/beads_priority.md` — priority system community research

**CLI flag reference (confirmed by codebase research):**
- `bd create`: `--notes`, `--design`, `--acceptance`, `--spec-id`, `--external-ref`
- `bd update`: `--append-notes`, `--design`, `--acceptance`, `--spec-id`, `--add-label` (repeatable), `--remove-label`, `--claim`
- `bd init`: `--prefix` (auto-detects from dir name), `--team`, `--contributor`
- `bd doctor`: `--agent` (agent-facing output), `--fix`, `--json`

## Technical Considerations

- **Line budget:** ~500-550 lines. Current is 385. The reference.md template grows most.
  Compress existing prose where possible to make room.
- **SkillsBench constraint:** The onboard.md itself is a procedural command (linear steps),
  not a decision-making skill. Comprehensiveness is OK for procedures. But the *generated*
  reference.md is loaded by agents during work — it must be focused sections, not walls
  of text.
- **Plugin convention:** Unicode symbols, not emoji (per docs/PLUGIN.md).
- **Backward compatibility:** v3 onboard must work on projects that were onboarded by v2
  (no .onboard-state.yaml exists yet). Treat missing state file as "never onboarded or
  onboarded by older version."

### Step 1: Update Phase 0 — State Detection

**What changes:** Add three-way mode detection, auto-init flow, and re-onboard detection.

**Current Phase 0** (lines 12-28) checks: `.beads/` exists, config.yaml, conventions dir,
AGENTS.md markers, CLAUDE.md, doctor.

**Add after the `.beads/` check (item 1):**

```
If `.beads/` is missing:
  - Offer to run `bd init`: "No beads found. Initialize now?"
  - If yes: run `bd init --prefix <auto-detected-from-dir-name>`
  - Show what happened: "Initialized beads with prefix `<prefix>`"
  - Offer prefix customization: "Want a different prefix?"
  - Mention team mode exists: "For multi-user sync, re-run with `bd init --team`"
  - Continue to Phase 1 after init completes
  - If no: stop with message about needing beads initialized first
```

**Add after the conventions check (item 3), replace the current re-onboard logic:**

```
Read `.beads/conventions/.onboard-state.yaml` if it exists:
  - Extract: version, timestamp, answers
  - Compare version against current onboard version (embed a version constant, e.g., "v3")

Re-onboard behavior:
  - If no state file AND no conventions dir: first-time onboard (proceed normally)
  - If no state file BUT conventions dir exists: onboarded by older version (pre-state-file)
    → "Conventions exist but were generated by an older onboard. Upgrading to v3 adds
       description templates, priority calibration, and lifecycle guidance. Re-run
       interview with your existing answers as defaults, or start fresh?"
  - If state file exists AND version < current:
    → Same upgrade message, but pre-fill all interview questions from saved answers
  - If state file exists AND version = current:
    → "Conventions are current (v3). Review/update your answers, or start fresh?"
  - If conventions were manually edited (file content differs from what onboard would
    generate — detect by checking if the state file's answers would produce the current
    files): warn "Your convention files have custom edits. New guidance will be merged
    alongside your customizations, not overwrite them."
```

**Add project state detection (new item, after doctor):**

```
Detect project state:
  - If significant code exists (>5 source files, or package manager config):
    → mode = "existing"
  - Else if planning docs exist (check docs/brainstorms/, docs/plans/, docs/, root .md):
    → mode = "planned"
  - Else:
    → mode = "empty"
```

### Step 2: Update Phase 1 — Adaptive Detection Sources

**What changes:** Add fallback chain for detection and project type detection.

**Current Phase 1** (lines 29-49) does codebase analysis only.

**Replace with adaptive chain:**

```
Detection source depends on project state (from Phase 0):

If mode = "existing" (current behavior):
  1. Directory structure → area labels
  2. File extensions + package files → languages/frameworks
  3. Git log (last 20 commits) → commit conventions
  4. .beads/config.yaml → issue prefix
  5. AGENTS.md / CLAUDE.md → existing content

If mode = "planned":
  1. Read planning docs (brainstorms, plans, specs in docs/)
  2. Extract: project description, planned file structure, tech stack mentions
  3. Map planned directories to candidate area labels
  4. Default commit format: type(scope): description (no history to detect from)
  5. .beads/config.yaml → issue prefix

If mode = "empty":
  1. No detection — use universal defaults
  2. Default areas: area:core (placeholder, user will customize)
  3. Default commit format: type(scope): description
  4. .beads/config.yaml → issue prefix

All modes — add project type detection:
  Detect from signals: deployment configs (Dockerfile, Vercel, Netlify, AWS) = production
  app. Library indicators (go.mod with module path, package.json with main/exports) =
  library. No deployment + no exports = personal/internal project. Store as project_type
  for priority calibration in Phase 3.
```

Generate three project descriptions that ground the project in reality, not vision.

**What changes:** Add 5 new sections to the reference.md template in Phase 3. This is the

**Add after the existing "Dependencies" section in the reference.md template:**

#### Section A: Description Structure by Issue Type

```markdown
## Description Structure

Structure your descriptions by issue type. Agents that don't know which sections
to include default to one-line descriptions — these are unsearchable and useless
for future sessions.

### Epic
Include when relevant: Context, Core Issues, Phases [NOW/SOON/LATER], Key Decisions,
Related beads

### Bug
Include when relevant: Observed behavior, Expected behavior, Steps to reproduce,
Environment

### Feature
Include when relevant: Motivation (why this matters), Acceptance criteria, Scope
boundary (what's NOT included)

### Task
Include when relevant: Context (why now, not later), Scope, Done-when

### Chore
Include when relevant: What + why now (not just "update deps" — why specifically)

### Examples

[Generate one complete `bd create` command per type using the project's prefix and
labels. Descriptions should be filled in with realistic content, not placeholder
text. These are the pattern-matching examples agents learn from.]

**Bug example:**
` ``
bd create --title="[realistic bug title]" --type=bug --priority=1 \
  --labels=area:[detected],ux \
  --description="Observed: [specific behavior]

Expected: [correct behavior]

Steps to reproduce:
1. [step]
2. [step]

Environment: [browser/OS/version]"
` ``

**Feature example:**
` ``
bd create --title="[realistic feature title]" --type=feature --priority=2 \
  --labels=area:[detected] \
  --description="Motivation: [why this matters for users/project]

Acceptance criteria:
- [concrete criterion 1]
- [concrete criterion 2]

Scope boundary: Does NOT include [explicit exclusion]"
` ``

[Generate similar for task, chore. Epic example should show the Phases structure.]
```

#### Section B: Priority Calibration

```markdown
## Priority Semantics

Priority determines creation-time classification. `bd ready` handles work ordering
via the dependency graph — priority is a tiebreaker when multiple unblocked items exist.

[Generate project-type-specific table based on Phase 1 project_type detection:]

**For production web apps:**
| Priority | Meaning | Example |
|---|---|---|
| P0 | Site down, users can't access, data loss | Database connection failures, auth broken |
| P1 | Core feature broken, major regression | Checkout flow error, search returns no results |
| P2 | Meaningful improvement or non-critical bug | Visual glitch on settings page, slow load |
| P3 | Minor polish, non-urgent | Tooltip wording, button alignment |
| P4 | Backlog, speculative | "Someday" ideas, exploratory |

**For CLI tools:**
| P0 | Data loss or corruption | Corrupts config file, deletes wrong data |
| P1 | Core command broken, blocking workflow | Main command crashes, output format broken |
| P2 | Confusing behavior or error message | Misleading error, unclear help text |
| P3 | Minor friction | Extra flag needed for common operation |
| P4 | Backlog | Nice-to-have features, exploratory |

[Include whichever table matches the detected project type. If unclear, include the
most relevant one.]

### Scope awareness

When creating beads, consider complexity alongside priority. A self-contained fix in
one file is a different bead from a multi-file refactor, even if they address the same
problem. Smaller, focused beads are easier for agents to complete in a single context
window. If a bead's scope exceeds what you can complete in your current session, break
it into sub-beads before starting.
```

#### Section C: bd create Fields

```markdown
## Create Fields Beyond --description

`bd create` supports fields that add structured, searchable context. Use them when
the trigger condition applies — skip them otherwise. An empty `--notes ""` is worse
than omitting the field.

| Field | Trigger | Decision Test | Example |
|---|---|---|---|
| `--description` | Always | — | Always include |
| `--notes` | Discovered something non-obvious | "Would a future agent hit this same surprise?" | `--notes="Redis pool must be >=5 for concurrent workers. Default of 3 causes intermittent failures."` |
| `--design` | Chose between alternatives | "Is there a 'why not the other way?' someone would ask?" | `--design="Redis pub/sub over WebSockets — deployment has no sticky sessions"` |
| `--acceptance` | Done-when isn't obvious from title | "Could two agents disagree on whether this is complete?" | `--acceptance="Lighthouse >=90, loads <2s on 3G throttle, works in Safari 16+"` |
| `--spec-id` | A planning doc drives this work | "Is there a doc explaining why this bead exists?" | `--spec-id="docs/plans/2026-01-auth-redesign.md"` |
```

#### Section D: Bead Lifecycle — bd update During Work

```markdown
## Enriching Beads During Work

Don't wait until creation and close to add context. Use `bd update` at specific
moments during implementation to build structured, searchable metadata.

| Moment during work | Action | Instead of |
|---|---|---|
| Made a design decision | `bd update <id> --design "chose X because Y"` | Commenting "decided to use X because Y" |
| Found an edge case or gotcha | `bd update <id> --append-notes "edge case: Z"` | Commenting "note: also affects Z" |
| Realized what "done" means changed | `bd update <id> --acceptance "new criteria"` | Commenting "realized we also need Y" |
| Created a related doc | `bd update <id> --spec-id "docs/..."` | Commenting "see docs/plans/foo.md" |
| Scope became clearer | `bd update <id> --add-label concern` | Not updating labels at all |

The pattern: if the information describes **the bead itself** (what it is, how to
build it, when it's done), use a field. If it describes **what happened while working**
(progress, questions, handoffs), use a comment.
```

#### Section E: Comments vs Fields

```markdown
## When to Use Comments vs Fields

> **Decision test:** Would this information still be useful if all comments were deleted?
> If yes — it belongs in a structured field, not a comment.

### Comments are right for:
- Session handoff notes: "Stopped at step 3, X works, Y needs Z"
- Questions or blockers: "Need clarification on IE11 support"
- Progress checkpoints: "Phase 1 complete, starting Phase 2"
- External references: "Related: github.com/..."
- Post-close addenda: "Found edge case after closing, see [prefix]-xxx"

### Comments are wrong for:
| Instead of this comment... | ...use this update |
|---|---|
| "Decided to use Redis because no sticky sessions" | `bd update <id> --design "Redis — no sticky sessions in deployment"` |
| "Note: config falls back to env vars in Docker" | `bd update <id> --append-notes "Config falls back to env vars when no file (Docker)"` |
| "Done when Lighthouse >= 90" | `bd update <id> --acceptance "Lighthouse >= 90, <2s on 3G"` |
```

#### Section F: Context Window Awareness (brief note)

```markdown
## Session Scope

Consider context window scope when claiming work. A self-contained bug fix in one
file is completable in a single session. A refactor touching 15 files may need to
be broken into sub-beads or may benefit from worktrees/subagents. If a bead's scope
exceeds what you can complete in your current context window, break it down before
starting.
```

#### Section G: Convention Health Check (brief note)

```markdown
## Convention Health

Periodically audit your beads quality: Are descriptions structured? Are close reasons
rich? Are labels consistent? If conventions feel stale or agents aren't following them,
re-run `/beads:onboard` to upgrade with improved guidance.
```

## Dependencies and Risks

**Dependencies:**
- On `feature/onboard-command` branch (existing v2 implementation)
- No Go code changes needed — all CLI flags already exist

**Risks:**
- reference.md template may be too long, causing agents to skip sections. Mitigated
  by keeping each section focused with clear headers (SkillsBench: focused > comprehensive).
- Re-onboard "manual edit detection" is hard to implement perfectly in a markdown skill.
  The LLM reads the files and uses judgment — acceptable for v3, refine based on feedback.
- Line budget may be tight (~500-550). If it goes over, compress existing prose first
  before cutting new content.
